# Adding a Second NVMe Scratch Disk to an AMD AI Workstation

This note documents the installation and configuration of a second NVMe
SSD in an AMD-based AI workstation used for Linux, ROCm, and local AI
workflows.

The goal was to separate:
- the primary system / model storage
from:
- scratch space
- temporary files
- dataset staging

This is a common and practical optimization for AI workstations because
it reduces I/O contention during model loading, preprocessing, inference,
and training.


2. Secondary SSD added:
   
GOODRAM PX700 1 TB NVMe
PCIe 4.0 x4
M.2 2280

3. STORAGE LAYOUT TARGET

Primary NVMe (4 TB):
- system
- boot
- home
- models
- checkpoints

Secondary NVMe (1 TB):
- /scratch
- temporary AI files
- dataset staging
- cache / work area

Planned logical layout:

NVMe 4 TB  -> /, /boot/efi, models, main storage
NVMe 1 TB  -> /scratch


4. WHY ADD A SECOND NVME

For AI workloads this layout is useful because:
- system traffic does not compete with scratch writes
- temporary files can be isolated
- dataset operations are cleaner
- cache and preprocessing steps can be redirected away from the system disk
- workstation organization is improved

In practice, this is a sensible configuration for ROCm, PyTorch,
Transformers, vLLM, and other local AI workflows.


5. PHYSICAL INSTALLATION

The 1 TB GOODRAM PX700 NVMe SSD was installed in the secondary M.2 slot.

Key practical notes:
- the motherboard detected the drive correctly
- Linux detected the new NVMe device without additional drivers
- the drive required partitioning and formatting before use

6. DEVICE DETECTION IN LINUX

lsblk

Observed result:
- nvme1n1 -> main 4 TB system drive
- nvme0n1 -> newly added ~953 GB drive

This confirmed that:
- the BIOS / UEFI recognized the SSD
- Linux kernel recognized the SSD
- PCIe connectivity was working correctly

```
sudo apt install nvme-cli

sudo nvme list
```

Observed output identified both devices, including:
- SSDPR-PX700-01T-80
- SSDPR-PX700-04T-80

This confirmed correct model identification at the operating system level.


7. PARTITIONING THE NEW DRIVE

The new drive was partitioned with fdisk:
```
sudo fdisk /dev/nvme0n1
```
Inside fdisk:
- create GPT label
- create a new partition
- write changes


Equivalent interactive steps:
```
g
n
Enter
Enter
Enter
w
```

Result:
- /dev/nvme0n1p1 created successfully


8. FORMATTING THE NEW PARTITION

The new partition was formatted as ext4:
```
sudo mkfs.ext4 /dev/nvme0n1p1
```

Result:
- filesystem created successfully
- UUID assigned to the partition

Confirmed UUID from blkid: 6526736f-4bcc-4cab-b56a-9c6182781a8f


9. MANUAL MOUNTING

A scratch directory was created:
```
sudo mkdir /scratch
```
Then the new partition was mounted manually:
```
sudo mount /dev/nvme0n1p1 /scratch
```
Verification with lsblk showed:
```
nvme0n1
└─nvme0n1p1   mounted at /scratch
```

This confirmed that:
- partitioning succeeded
- formatting succeeded
- mount point worked correctly


10. PERSISTENT MOUNT VIA /etc/fstab

To mount the scratch disk automatically after reboot, an entry was added
to /etc/fstab.

Command used to inspect the correct UUID:
```
sudo blkid
```
Correct /etc/fstab entry:

UUID=6526736f-4bcc-4cab-b56a-9c6182781a8f /scratch ext4 defaults,noatime,nodiratime 0 2

```
sudo systemctl daemon-reload
sudo mount -a
```

No error is returned, the configuration is valid.


11. FINAL VERIFIED STATE

/dev/nvme1n1
- main 4 TB drive
- system disk
- contains / and /boot/efi

/dev/nvme0n1
- 1 TB GOODRAM PX700
- formatted ext4
- mounted as /scratch

This is the intended and working result.


12. WHY /SCRATCH IS A GOOD CHOICE FOR AI

Using /scratch on a separate NVMe is a practical design choice because it
can be used for:
- temporary preprocessing files
- dataset unpacking
- model conversion artifacts
- benchmark outputs
- large intermediate files
- temporary cache during experimentation

This makes the main system disk cleaner and reduces the risk of filling
the root partition during heavy experimentation.


13. SUGGESTED AI ENVIRONMENT VARIABLES

For AI workflows, the following idea is recommended:
```
TMPDIR=/scratch
```
Additional cache paths can also be redirected as needed, for example:
- Hugging Face cache
- temporary benchmark files
- dataset extraction paths

Example concepts:
```
export TMPDIR=/scratch
export HF_HOME=/scratch/hf
export TRANSFORMERS_CACHE=/scratch/hf
```
