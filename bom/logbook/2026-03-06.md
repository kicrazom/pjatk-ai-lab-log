1. OpenGL Renderer Verification

Command:
glxinfo | grep "OpenGL renderer"

Result:
OpenGL renderer string: AMD Ryzen 9 9950X3D (radeonsi, raphael_mendocino)

Conclusion:
The desktop session was rendering on the integrated GPU (iGPU) instead of the dedicated Radeon GPU.

------------------------------------------------------------

2. GPU Detection and Driver Validation

Commands:
lspci | grep VGA
lspci -k | grep -EA3 'VGA|Display'

Result:
Kernel driver in use: amdgpu

Conclusion:
The amdgpu kernel driver is correctly loaded and the Radeon GPU is detected.

------------------------------------------------------------

3. Vulkan Device Detection

Command:
vulkaninfo | grep deviceName

Detected devices:
- AMD Ryzen iGPU
- AMD Radeon (gfx1201)
- llvmpipe (software renderer)

Conclusion:
The Vulkan stack (RADV) is functioning correctly and detects the discrete GPU.

4. PCIe Link Verification

Command:

Speed 32GT/s
Width x16

The GPU is operating at full PCIe x16 bandwidth.

Command:

Renderer reported:
GL_RENDERER = AMD Radeon Graphics (radeonsi, gfx1201)

Conclusion:
OpenGL rendering works correctly on the discrete GPU when forced via DRI_PRIME.

7. Weasis DICOM Viewer Rendering Issue

Observed problem:
Gray screen during OpenGL rendering of DICOM data.

Exception:
Exception in thread "Texture 3D loader (OpenGL)"
java.lang.IllegalStateException:
Segment is too large to wrap as ByteBuffer
Size: 2618164224

Analysis:
The issue is related to:
- Java OpenGL pipeline
- GPU selection
- Large volumetric dataset (>2GB texture buffer)

This is not a driver or Mesa issue.

------------------------------------------------------------

8. Working Launch Configuration for Weasis

Command:
DRI_PRIME=1 JAVA_TOOL_OPTIONS="-Dsun.java2d.opengl=false" weasis -Xmx16G

Effect:
- Forces rendering on the Radeon GPU
- Disables problematic Java2D OpenGL pipeline
- Increases JVM memory allocation

Result:
Weasis renders large DICOM datasets correctly.
