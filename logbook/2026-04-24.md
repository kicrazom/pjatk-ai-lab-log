# Logbook — 2026-04-24

## Qwen 2.5 72B AWQ TP=2 sanity load and hypothesis testing on 2× R9700

### Summary

First successful dual-GPU tensor-parallel inference on the workstation for a
model that does not fit on a single 32 GiB R9700: Qwen 2.5 72B AWQ loaded with
`tensor_parallel_size=2`, generated coherent output, shutdown cleanly. Single
request throughput is 30× below the 150-400 tok/s target stated for this
configuration in benchmarks/results/README.md. Six configurations tested
systematically (H0, H1, H2, H3, H4, H6) to identify the bottleneck.
Best-performing configuration: **H3** (`enforce_eager=True` alone, without
`AMD_SERIALIZE_KERNEL` / `HIP_LAUNCH_BLOCKING`) at 5.0 tok/s per request.

Single session, ~3 hours end-to-end.

### Milestone

2× R9700 (gfx1201, RDNA 4) validated for tensor-parallel inference on a model
that exceeds single-card VRAM. Per-GPU state after load:

- Model weights: 21.63 GiB
- KV cache: 4.89 GiB → 32,048 tokens
- Max concurrency at 4k tokens per request: 7.82×
- Idle-to-generation draw: ~60 W → ~220 W per GPU (live `rocm-smi`)
- Symmetric load across cards (VRAM and power within 2% between GPU 0 and 1)

Model output quality is coherent in both English and Polish at all tested
configurations (sample prompts: quantum entanglement explanation, haiku,
capital of Poland, open science benefits).

### Hypothesis testing — six configurations

Each test: fresh `vllm.LLM(...)` from scratch, 30s cooldown between runs,
same 4 prompts, same sampling params, separate log file per hypothesis.

#### vLLM spawn multiprocessing fix (required before any test could run)

First orchestration attempt failed on all four tests with:

```
An attempt has been made to start a new process before the
current process has finished its bootstrapping phase.
```

Root cause: the parametrized sanity script placed `from vllm import LLM` and
`LLM(...)` at module level. vLLM with `tensor_parallel_size ≥ 2` forces the
`spawn` multiprocessing start method (because CUDA is initialized in the
parent), which re-imports the entire module in each worker — causing
recursive spawn failures. The original `test_dual_gpu_tp.py` wraps this
correctly in `def main()` and worked.

Fix: wrap all CUDA-using code, including `from vllm import ...`, inside a
function called only from `if __name__ == "__main__":`. Script refactored
into `build_vllm_kwargs()`, `print_environment()`, `run_sanity()`, `main()` —
SRP-compliant with type hints.

#### H4 — cache warm: REJECTED

Second run with identical flags. Load time collapsed 300s → 78s (vLLM
metadata cache effective between sessions). Warmup and generation remained
at ~95s — Triton kernels recompile each session regardless of cache.
Throughput 3.2 vs H0's 4.6 is within noise for 4-prompt runs. Cold compile
is not the dominant cost.

#### H3 — removing SERIALIZE + LAUNCH_BLOCKING: +56% confirmed

Unset both `AMD_SERIALIZE_KERNEL` and `HIP_LAUNCH_BLOCKING`. Result: 3.2 →
**5.0 tok/s**, warmup 6s faster. Real improvement, far from sufficient.

**Interpretation requires rereading the April 17 workaround.** At the time
all three variables — `enforce_eager`, `SERIALIZE_KERNEL=3`,
`HIP_LAUNCH_BLOCKING=1` — were added together as a single workaround for
`HSA_STATUS_ERROR_INVALID_PACKET_FORMAT`. They are not independently
motivated debug flags; `SERIALIZE_KERNEL` and `LAUNCH_BLOCKING` force
synchronous kernel execution, which avoids async-timing bugs that trigger
the HSA error at capture. The April 17 protocol used belt-and-suspenders.

H3 demonstrates that **on Qwen 72B AWQ with `enforce_eager=True`, the other
two flags are redundant** — no HSA error occurs, and removing them saves
36% of generation time. This doesn't invalidate the original workaround
(which covered graph capture too), but narrows it: eager alone is
sufficient for inference once capture is disabled.

#### H2 — `enforce_eager=False`: crash as expected

Disabled eager. vLLM attempted graph capture (`cudagraph_mode=FULL_AND_PIECEWISE`).
Capture completed (KV cache dropped 32,048 → 29,248 tokens; graph allocation
6.56 GiB; load 198s). First warmup forward pass failed:

```
HSA_STATUS_ERROR_INVALID_PACKET_FORMAT: The AQL packet is malformed. code: 0x1009
torch.AcceleratorError: HIP error: unspecified launch failure
```

Engine core died, both workers shut down. **Confirms the April 17 observation
that `enforce_eager=True` is mandatory on gfx1201 with vLLM 0.19.0+rocm721.**
CUDA graphs are still unusable on RDNA 4 as of today. Worth retesting after
the next vLLM release.

#### H1 — explicit `quantization="awq_marlin"`: no effect

3.2 tok/s, identical to H4. Inspection of vLLM logs confirms awq_marlin is
already auto-selected:

```
INFO awq_marlin.py:245: The model is convertible to awq_marlin during runtime. Using awq_marlin kernel.
INFO awq_marlin.py:408: Using ConchLinearKernel for AWQMarlinLinearMethod
```

The selected kernel is `ConchLinearKernel` (from `conch-triton-kernels
1.2.1`) — a generic Triton-based path, not an AMD-native matmul. Explicit
flag is redundant.

#### H6 — `VLLM_ROCM_USE_AITER=1`: no effect on AWQ path

Motivated by the observation that `amd-aiter 0.1.10.post2` is installed and
explicitly supports gfx1201:

```python
# aiter/jit/utils/chip_info.py
16: "gfx1200",  17: "gfx1201",
# aiter/ops/triton/.../utils.py
RDNA_ARCHS = frozenset({..., "gfx1200", "gfx1201"})
```

AITER is the master switch for several ROCm-native kernel families
(`AITER_LINEAR`, `AITER_MHA`, `AITER_RMSNORM`, `AITER_MoE`, all default True
once the master is enabled). Enabling it should engage native AMD kernels
instead of the generic Triton fallback.

Result: **3.4 tok/s**, within noise of H3.

Likely reason: `vllm/model_executor/kernels/linear/scaled_mm/aiter.py`
targets Int8 W8A8 (`AiterInt8ScaledMMLinearKernel`), not AWQ W4A16. For
AWQ the effective kernel path remains `awq_marlin → ConchLinearKernel`
regardless of the AITER master switch. AITER on this workload is dead code.

**Note:** AITER may still matter for BF16/FP16 models in the benchmark set
(Bielik 11B FP16, PLLuM 12B, Mistral-Nemo, PLLuM 70B triplet). The `AITER=0`
setting from the April 17 protocol should be re-validated per model.

### Lessons learned

- **Re-read the April 17 workaround before changing anything.** The three
  environment variables (`VLLM_ROCM_USE_AITER=0`, `AMD_SERIALIZE_KERNEL=3`,
  `HIP_LAUNCH_BLOCKING=1`) were added together, as a single workaround
  for HSA errors during graph capture. They look like independent debug
  flags but are historically coupled. H3 narrows the workaround:
  `enforce_eager=True` alone is sufficient once graph capture is disabled.

- **vLLM TP ≥ 2 spawn rule.** Any parametrized launcher script for
  multi-GPU vLLM must wrap the `LLM(...)` construction inside a function
  called only from the `__main__` guard. Module-level CUDA initialization
  causes `spawn` workers to re-execute the init path and fail. The
  existing `test_dual_gpu_tp.py` pattern is the correct one to copy.

- **100% GPU utilization is not proof of good throughput.** Both R9700
  cards sat at 100% during the 5 tok/s run. The compute was happening
  but in a slow code path (generic Triton, not AMD-native). GPU use %
  tells you nothing about whether the kernel is well-optimized for this
  architecture. Power draw is a better proxy (220 W per GPU here vs the
  ~300 W TDP headroom suggests the cards are not heat-limited).

- **AITER on AWQ is dead code in this stack.** AITER's scaled-MM kernel
  targets Int8, not AWQ W4A16. Enabling the master switch has no effect
  on AWQ throughput. Revisit for non-quantized and Int8 models in the
  benchmark set.

- **Cached ≠ warm.** vLLM caches engine metadata between sessions (load
  time 300s → 78s), but Triton kernel compilation still happens at
  warmup every session (~90s regardless of cache state). For pilot
  timing, expect 76s load + 91s warmup per run = ~170s before the
  actual benchmark starts.

### Key findings

**Plateau at ~121 tok/s for N≥500.** Throughput saturates between N=500
and N=1000 with less than 0.2% change despite the batch doubling. Total
time scales linearly with N at plateau (526s → 1051s), meaning additional
requests queue rather than parallelize beyond what the scheduler already
maintains.

**Linear scaling up to N=100.** From N=10 to N=100 the throughput grows
approximately linearly with a ~8× ratio, after which the growth rate
collapses sharply. The transition occurs between N=100 (110.1 tok/s) and
N=200 (117.6 tok/s) where the next doubling adds only 7% throughput.

**No preemption observed at N=1000.** The N=1000 run completed in 17.5
minutes within the 30-minute subprocess timeout. GPU memory utilization
stayed below full during peak (26.3 GiB of 31.86 GiB per card observed
in dashboard), so KV cache pressure did not trigger preemption events
visible in the logs.

| N    | Qwen 7B TP=2 (Plan A) | Qwen 72B AWQ TP=2 (Plan B) | Ratio |
|-----:|----------------------:|---------------------------:|------:|

- TP=2 all-reduce cost proportionally higher for 72B (80 layers) vs. 7B

nightly stack (planned as Phase D).

## CCD pinning — single NUMA observation

The workstation CPU is an AMD Ryzen 9 9950X3D with two CCDs:

- **CCD 0** (die:0): logical CPUs 0-7 + SMT 16-23, 32 MB L3, max boost ~5.7 GHz
- **CCD 1** (die:1): logical CPUs 8-15 + SMT 24-31, **96 MB L3 with 3D V-Cache**, max boost ~5.2 GHz

`numactl --hardware` reports a single NUMA node (`available: 1 nodes (0)`),
so cross-CCD access is not a NUMA hop — it traverses Infinity Fabric within
the same node. The GPUs report `NUMA: -1` in sysfs, which on single-node
systems means "no preference" rather than invalid. During Plan B, the
default worker affinity was the full CPU set (0-31) and kernel scheduler
placed both `VLLM::Worker_TP0` and `VLLM::Worker_TP1` on CCD 1 cores at
the moments sampled from the dashboard.

Hypothesis: forcing both workers to CCD 1 (to maximize shared 3D V-Cache
locality and eliminate cross-CCD migration) might produce a measurable
improvement. Tested at N=100 with:

```
taskset -c 8-15,24-31 python bench_with_thermals_qwen72b.py 2 100 ...
```

Worker affinity confirmed with `taskset -cp $worker_pid`: both workers
reported `lista przypisań: 8-15,24-31`, confirming the affinity mask
propagated through vLLM's spawn-multiprocessing.

### Result

| Config | Total time | Output throughput |
|--------|-----------|-------------------|
| Plan B (default affinity 0-31) | 116.81s | 110.1 tok/s |
| Smoke H7 (default affinity 0-31) | 116.21s | 110.1 tok/s |
| Smoke CCD1 (affinity 8-15,24-31) | 116.22s | 110.1 tok/s |

Three identical measurements to three significant figures.

### Conclusion

CCD pinning has no measurable effect on Qwen 72B AWQ TP=2 throughput on
this hardware. Three independent runs at the same N=100 produced the same
110.1 tok/s, confirming the benchmark's determinism (same seed, same
prompts, same kernels) and independently confirming that CPU affinity is
not a lever here. The workload is GPU-bound (100% utilization on both
cards throughout the run) with total CPU utilization around 14% (~4.5 cores
busy from the available 32). The 3D V-Cache of CCD 1 cannot accelerate
what the CPU is not a bottleneck for.

Negative result, but a clean one — worth documenting as a calibration
point for future experiments that might face similar questions. NUMA-style
CPU pinning is not a productive direction for dense 70B-class AWQ models
on a single-socket Ryzen workstation.

### Methodology comparison

Key differences between our Plan B protocol and kyuz0's
`/opt/run_vllm_bench.py`:

| Parameter                      | Our Plan B                    | Kyuz0 `run_vllm_bench.py` |
|--------------------------------|-------------------------------|---------------------------|
| Benchmark entry point          | Direct `LLM()` + `generate()` | `vllm bench throughput` CLI |
| Dataset                        | 8 templates × 20 topics       | ShareGPT (real conversations) |
| Prefix caching impact          | Very high (templated)         | Low (unique prompts) |
| Output length                  | 128 tokens fixed              | 512 tokens forced |
| Input length                   | ~13 tokens mean               | ~1000 tokens (ShareGPT) |
| `--num-prompts`                | 1000 (at peak sweep point)    | 1000 |
| `--gpu-memory-utilization`     | 0.92                          | 0.98 |
| `--max-num-seqs`               | default (256)                 | 32 |
| `--max-num-batched-tokens`     | default                       | model-specific, e.g. 16384 |
| CUDA graphs                    | Disabled (`enforce_eager=True`) | Enabled (`enforce_eager=False`) |
| Quantization env               | default                       | `VLLM_USE_TRITON_AWQ=1` for AWQ models |

These methodology differences are substantial. A fair apples-to-apples
comparison with kyuz0's published numbers requires adopting his protocol,
not just switching stacks. In particular, the 4× longer output length
(128 → 512) amortizes prefill cost across more decode steps and
typically shows measurably higher tokens/sec for the same stack.

Kyuz0's published throughput for the structurally closest model to ours
(`Qwen3-Next-80B-A3B-Instruct-AWQ-4bit`, TP=2, Triton backend) is **680.23
tok/s**. Caveat: that model is MoE with A3B (3B active parameters out of
80B). Per-token memory traffic is approximately 1.5 GB (4-bit × 3B
active), versus 36 GB for our Qwen 2.5 72B dense AWQ. A 24× reduction
in memory traffic explains most of the 5.8× throughput advantage kyuz0
reports for that model class alone, before factoring in the stack or
methodology differences. The only directly comparable dense model in
kyuz0's table is Gemma-3-27B FP8 dense TP=2 at 984 tok/s; scaling that
naively to 72B (memory-bound) gives `984 × (27/72)² ≈ 138 tok/s`, which
places our measured 121 tok/s within the expected range for this class
of model on this class of hardware.

## Conclusions for project goal A (hardware ceiling)

For **Qwen 2.5 72B dense AWQ on 2× R9700 TP=2 with the stable stack**,

## CCD pinning — single NUMA observation

The workstation CPU is an AMD Ryzen 9 9950X3D with two CCDs:

- **CCD 0** (die:0): logical CPUs 0-7 + SMT 16-23, 32 MB L3, max boost ~5.7 GHz
- **CCD 1** (die:1): logical CPUs 8-15 + SMT 24-31, **96 MB L3 with 3D V-Cache**, max boost ~5.2 GHz

`numactl --hardware` reports a single NUMA node (`available: 1 nodes (0)`),
so cross-CCD access is not a NUMA hop — it traverses Infinity Fabric within
the same node. The GPUs report `NUMA: -1` in sysfs, which on single-node
systems means "no preference" rather than invalid. During Plan B, the
default worker affinity was the full CPU set (0-31) and kernel scheduler
placed both `VLLM::Worker_TP0` and `VLLM::Worker_TP1` on CCD 1 cores at
the moments sampled from the dashboard.

Hypothesis: forcing both workers to CCD 1 (to maximize shared 3D V-Cache
locality and eliminate cross-CCD migration) might produce a measurable
improvement. Tested at N=100 with:

```
taskset -c 8-15,24-31 python bench_with_thermals_qwen72b.py 2 100 ...
```

Worker affinity confirmed with `taskset -cp $worker_pid`: both workers
reported `lista przypisań: 8-15,24-31`, confirming the affinity mask
propagated through vLLM's spawn-multiprocessing.

### Result

| Config | Total time | Output throughput |
|--------|-----------|-------------------|
| Plan B (default affinity 0-31) | 116.81s | 110.1 tok/s |
| Smoke H7 (default affinity 0-31) | 116.21s | 110.1 tok/s |
| Smoke CCD1 (affinity 8-15,24-31) | 116.11s | 110.2 tok/s |

Three independent measurements, maximum spread 0.1 tok/s
(±0.09%), well within benchmark run-to-run variance.

### Conclusion

CCD pinning has no measurable effect on Qwen 72B AWQ TP=2 throughput on
this hardware. Three independent runs at the same N=100 produced results
within 0.1 tok/s of each other (110.1, 110.1, 110.2), well within
benchmark noise, confirming both the benchmark's determinism (same seed,
same prompts, same kernels) and independently confirming that CPU
affinity is not a lever here. The workload is GPU-bound (100% utilization
on both cards throughout the run) with total CPU utilization around 14%
(~4.5 cores busy from the available 32). The 3D V-Cache of CCD 1 cannot
accelerate what the CPU is not a bottleneck for.

Negative result, but a clean one — worth documenting as a calibration
point for future experiments that might face similar questions. NUMA-style
CPU pinning is not a productive direction for dense 70B-class AWQ models
on a single-socket Ryzen workstation.

### Methodology comparison

Key differences between our Plan B protocol and kyuz0's
`/opt/run_vllm_bench.py`:

| Parameter                      | Our Plan B                    | Kyuz0 `run_vllm_bench.py` |
|--------------------------------|-------------------------------|---------------------------|
| Benchmark entry point          | Direct `LLM()` + `generate()` | `vllm bench throughput` CLI |
| Dataset                        | 8 templates × 20 topics       | ShareGPT (real conversations) |
| Prefix caching impact          | Very high (templated)         | Low (unique prompts) |
| Output length                  | 128 tokens fixed              | 512 tokens forced |
| Input length                   | ~13 tokens mean               | ~1000 tokens (ShareGPT) |
| `--num-prompts`                | 1000 (at peak sweep point)    | 1000 |
| `--gpu-memory-utilization`     | 0.92                          | 0.98 |
| `--max-num-seqs`               | default (256)                 | 32 |
| `--max-num-batched-tokens`     | default                       | model-specific, e.g. 16384 |
| CUDA graphs                    | Disabled (`enforce_eager=True`) | Enabled (`enforce_eager=False`) |
| Quantization env               | default                       | `VLLM_USE_TRITON_AWQ=1` for AWQ models |

These methodology differences are substantial. A fair apples-to-apples
comparison with kyuz0's published numbers requires adopting his protocol,
not just switching stacks. In particular, the 4× longer output length
(128 → 512) amortizes prefill cost across more decode steps and
typically shows measurably higher tokens/sec for the same stack.

Kyuz0's published throughput for the structurally closest model to ours
(`Qwen3-Next-80B-A3B-Instruct-AWQ-4bit`, TP=2, Triton backend) is **680.23
tok/s**. Caveat: that model is MoE with A3B (3B active parameters out of
80B). Per-token memory traffic is approximately 1.5 GB (4-bit × 3B
active), versus 36 GB for our Qwen 2.5 72B dense AWQ. A 24× reduction
in memory traffic explains most of the 5.8× throughput advantage kyuz0
reports for that model class alone, before factoring in the stack or
methodology differences. The only directly comparable dense model in
kyuz0's table is Gemma-3-27B FP8 dense TP=2 at 984 tok/s; scaling that
naively to 72B (memory-bound) gives `984 × (27/72)² ≈ 138 tok/s`, which
places our measured 121 tok/s within the expected range for this class
of model on this class of hardware.
