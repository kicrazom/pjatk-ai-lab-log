### gfx1201 workaround — enforce_eager=True

First forward pass reliably failed with:

```
HSA_STATUS_ERROR_INVALID_PACKET_FORMAT
(AQL packet malformed at async_copy_ready_event.synchronize)
```

The failure happens during CUDA-graph capture on gfx1201 with this vLLM
build. Workaround:

- `enforce_eager=True` in `LLM(...)` constructor
- Environment variables:
  - `VLLM_ROCM_USE_AITER=0`
  - `AMD_SERIALIZE_KERNEL=3`
  - `HIP_LAUNCH_BLOCKING=1`
- Clear compile cache on config changes: `rm -rf ~/.cache/vllm/torch_compile_cache/`

Performance cost of eager mode estimated at 20-30% vs graph capture, but
non-functional graphs are worse than slower eager.

**To re-test after each vLLM release** — this may be fixed upstream.

### Thermal instrumentation toolkit

Built three scripts to capture per-run thermal and utilization timelines:

- `sample_system.py` — background sampler (1 Hz) via psutil + rocm-smi --json.
  Captures CPU Tctl, per-core utilization, per-GPU temperature (edge/junction
  /memory), power, VRAM, and usage percentage. Writes JSONL. Correctly
  distinguishes discrete GPUs from the Raphael iGPU by VRAM size and name
  substring.
- `plot_thermals.py` — matplotlib renderer for JSONL + events.json.
  Two-panel chart: temperature timeline (GPU 0, GPU 1, iGPU, CPU Tctl) on top,
  utilization on bottom. Marks bench start/end.
- `bench_with_thermals.py` — wrapper orchestrating sampler start, 5s baseline,
  benchmark run, 5s cooldown, sampler stop, event log dump, plot generation.

Sanity test (TP=1, N=50) passed — sampler captures 44 points over 44s,
events.json has both markers, PNG generates correctly, bench throughput
matches earlier exploratory runs within noise (1068 tok/s vs 1054 tok/s).

### Plan A — full concurrency sweep

Executed `run_plan_a.sh` — orchestrator script for 7 concurrency levels x
2 TP configs = 14 runs, 45s cooldown between runs.

**Configuration (per run):**

- Model: Qwen/Qwen2.5-7B-Instruct
- dtype=float16, max_model_len=4096, gpu_memory_utilization=0.70
- enforce_eager=True
- max_tokens=128
- Prompts share 8 template prefixes (prefix caching active)

**Total sweep time: 29 min** (started 21:50:48, ended 22:20:22).

### Results

| N | TP=1 tok/s | TP=2 tok/s | TP=2 / TP=1 |
|---|---|---|---|
| 50 | 1057 | **1154** | 1.09x (TP=2 wins) |
| 100 | 1787 | 1772 | 0.99x |
| 200 | 3327 | 2628 | 0.79x |
| 500 | 3866 | 2938 | 0.76x |
| 1000 | 3863 | 2946 | 0.76x |
| 2000 | 3880 | 2942 | 0.76x |
| 3000 | 3848 | 2917 | 0.76x |

**Key findings:**

1. **Two flat plateaus:** TP=1 saturates at ~3870 tok/s from N=500 upward
   (std dev 0.3%). TP=2 saturates at ~2940 tok/s from N=500 upward (std dev
   0.5%). Both are remarkably stable over 500 to 3000 concurrent requests.

2. **24% PCIe tax for TP=2.** The 930 tok/s gap is all_reduce synchronization
   overhead on PCIe 5.0 x16 without NVLink. Since the 7B model fits on a
   single 32 GB R9700 with 6.5 GiB KV cache headroom, TP=2 provides no
   benefit here — only overhead.

3. **TP=2 wins only at N=50** (+9%). Small batches benefit from parallel
   prefill before all_reduce cost starts dominating. At every higher
   concurrency tested, TP=1 is faster.

4. **No KV pressure at N=3000.** Prefix caching with only 8 unique template
   prefixes keeps the effective token budget well under TP=1's 120k limit.
   To see TP=2's larger KV cache (487k tokens) provide value, need either
   unique prompts or longer decode.

### Thermal observations (from tp2-n3000, 131s compute)

- **GPU 0 runs 5C hotter than GPU 1** (peak 66C vs 61C). Same workload
  (tensor parallelism splits layer weights evenly), so the asymmetry is
  airflow / case position, not compute distribution. Worth investigating
  physical mitigations.
- **Both GPUs well below throttle.** Peak temps 30C below the 95C typical
  throttle. 45s inter-run cooldown is more than sufficient.
- **CPU is thermally dominant.** AMD 9950X3D Tctl hits 75C during compute,
  hotter than either GPU. Tokenization, HTTP scheduling, and NCCL bookkeeping
  on CPU at ~16-20% utilization generate significant heat.
- **iGPU spikes from dashboard.** Periodic 80-99% iGPU utilization at ~15s
  intervals correspond to the ai-workstation-dashboard WebSocket updates
  and browser rendering. No impact on benchmark (iGPU not in compute path)
  but visible in thermal timelines.

### Lessons learned

- **Version discipline matters.** The initial confusion between RDNA 3 and
  RDNA 4 would have led to wrong wheel selection. Always verify via
  `torch.cuda.get_device_properties().gcnArchName` rather than trusting
  manufacturer naming.
- **venv drops silently between sessions.** Repeated `ModuleNotFoundError:
  No module named 'vllm'` happened whenever a new terminal was opened or
  `pkill` killed a shell. Discipline: always prepend
  `source ~/venvs/vllm/bin/activate` and the three env vars to every
  bash session that touches vLLM.
- **Prefix caching dominates throughput for templated prompts.** Our
  workload with 8 shared prefixes shows very different scaling than a
  realistic production workload would. Throughput numbers here are upper
  bounds for TP ratio comparison, not absolute production numbers.
- **Position matters for cooling.** Two identical cards in the same case
  can differ by 5C under identical load. Thermal monitoring per-GPU is
  worth the small tooling cost.
