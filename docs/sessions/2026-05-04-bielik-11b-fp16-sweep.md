## Methodological humility (METHODOLOGY §8)

We measure inference *throughput*, *latency*, *thermal envelope*, and
*power efficiency* under varying concurrent load. We do not measure
model quality, reasoning capability, factual accuracy, or downstream
clinical utility. Following Lerchner (2026), these are extrinsic
computational properties of the inference vehicle, not constitutive
properties of cognition. Our claims terminate at the hardware-software
interface.

## Engineering observations (PUBLIC)

1. **Linear-then-saturating scaling.** Aggregate tok/s scales 89% efficient
   from N=10 to N=25, then enters sub-linear regime (~85% efficiency) up to
   N=200. Knee at N≈500 with plateau ~1794 tok/s. No abrupt collapse.

2. **Clean preemption.** N=500 vs N=1000: 1793.96 vs 1793.39 tok/s — drift
   0.03%. Wall time doubles (33.77s → 68.78s) but aggregate throughput is
   identical. vLLM's KV-cache swapping is working correctly: preemption
   trades latency for capacity without loss of throughput.

3. **Load time consistency.** Across all 7 runs, load time stayed in
   21.0-22.6s window. No memory leakage or fragmentation between runs
   despite full vLLM restart per N.

4. **No thermal throttling observed.** Temperatures stayed within nominal
   envelope throughout sweep. Detailed thermal traces in
   `thermal_gallery.png`.

## Engineering finding extending METHODOLOGY §3.2 (PUBLIC)

During Phase 1 envelope (2026-05-04), config
`bielik_11b_awq_tp1_max2048_util090_graphs` (i.e., `enforce_eager=False`)
hard-segfaulted in `libhsa-runtime64.so.1.18.70200`:

```
VLLM::EngineCor[28375]: segfault at 7b199c700000 ip 00007b1e5aa790c0
   sp 00007fffababeb00 error 4 in libhsa-runtime64.so.1.18.70200
   [790c0,7b1e5aa1d000+13c000]
```

This extends the constraint stated in METHODOLOGY §3.2 (which mentions only
hybrid-attention models like Qwen 3.5/3.6): **on gfx1201, even
Mistral-based models (no hybrid attention) require `enforce_eager=True`
because the CUDA-graph capture path crashes inside ROCm's HSA runtime.**

For Phase 2, all configs use `enforce_eager=True`. This was applied
preemptively to the FP16 TP=2 sweep based on the Phase 1 finding.

Recommendation: METHODOLOGY §3.2 errata for v1.1 — broaden the eager
requirement from "hybrid attention models" to "all models on gfx1201
under vLLM 0.19.0+rocm721 + PyTorch 2.10".

## Lessons learned (engineering)

- Sleep prevention via `systemd-inhibit` should wrap the orchestrator from
  the start, not be added mid-sweep. Today's sweep finished too fast for
  this to matter, but longer sweeps (Qwen 70B class) will need it.

- `tmux` was problematic on this workstation today (session exited
  prematurely on first attempt). For short sweeps (~10 min), running in
  foreground with `tee` to `/tmp` is simpler and equally reliable.

- 7-config envelope grid is sweet spot — fits METHODOLOGY §5.1's "8-12
  configs" budget while staying under 20 min wall time. Future models
  should use the same shape.
