## Abstract

Open-weight large language models in the 27 billion parameter range
have become accessible on consumer-grade GPUs as cards with 32 GB of
VRAM enter the market. We present empirical findings on running Qwen
3.6 27B (released 2026-04-22) on a dual-GPU workstation with two AMD
Radeon AI PRO R9700 cards (gfx1201, RDNA 4 / Navi 48) under ROCm 7.2.1
and vLLM 0.19.0. Five smoke-test configurations characterise the
practical envelope: tensor-parallel size (TP), CUDA-graph capture,
quantisation (FP8 vs BF16), memory utilisation budget, and maximum
context length. Two configurations reach inference; both require
TP=2 and `enforce_eager=True`. A counter-intuitive finding emerges
under the current software stack: BF16 outpaces FP8 by approximately
75% on first inference (7.23 vs 4.15 tok/s), traceable to the absence
of R9700-specific FP8 kernel configurations in vLLM. We document the
mandatory environment variables for stability, the precise memory
footprint per quantisation, and the implications for community
guidance on consumer RDNA 4 deployment. Full reproducibility material,
including the complete software environment manifest, is archived at
the project repository.

## 1. Introduction

The arrival of consumer-grade 32 GB GPUs from AMD's Radeon AI PRO
line, in particular the R9700 with the gfx1201 (RDNA 4 / Navi 48)
silicon, has shifted the boundary of what is locally deployable on
desktop hardware. Open-weight large language models in the 27–30
billion parameter range — previously requiring datacenter accelerators
or aggressive quantisation — now occupy a regime where two consumer
cards can plausibly host inference, given the right software stack.

The software side of that proposition has been less well-characterised.
vLLM, currently the dominant inference engine for production-quality
serving of open-weight models, has documented support for AMD
ROCm-based hardware. However, the published guidance focuses on
datacenter Instinct silicon (MI200 / MI300 family). The practical
envelope on consumer RDNA 4 hardware — what configurations load, what
configurations crash, what configurations are stable, what configurations
are merely slow — is not, to our knowledge, systematically documented
in the open literature as of April 2026.

This study addresses the gap for one specific model class: the Qwen
3.6 27B family released on 2026-04-22, available in both FP8 and BF16
variants. The 3.6 family inherits the 3.5 architecture's hybrid
Mamba + Transformer attention pattern, which is more
memory-fragmentation-prone than classical Transformer-only models and
exposes failure modes that simpler architectures may not exhibit.

Our contributions are five empirical findings, each with reproduction
material:

1. **Qwen 3.6 27B FP8 does not fit a single 32 GB R9700.** Under
   default vLLM configuration, weight padding alone exceeds the
   available memory budget.

2. **Default CUDA-graph capture crashes on gfx1201 with this model.**
   The crash mode is `HSA_STATUS_ERROR_INVALID_PACKET_FORMAT` at first
   inference, suggesting an AQL packet formatting issue specific to
   the hybrid attention pattern.

3. **`enforce_eager=True` is mandatory.** With eager mode, both FP8
   and BF16 variants reach inference under TP=2.

4. **A previously documented environment variable workaround
   `AMD_SERIALIZE_KERNEL=3` is silently ignored** by current PyTorch.
   The accepted values are 0 or 1; setting 3 leaves the workaround
   inactive without user notification.

5. **BF16 outpaces FP8 by approximately 75%** in cold first-inference
   throughput on this hardware, inverting the typical expectation
   from datacenter benchmarks.

Each finding is grounded in directly observed log output and is
falsifiable: a reader with comparable hardware can re-run the same
configurations and either confirm or contradict.

### 2.4 Test protocol

Five configurations were evaluated. For each, the first inference was
prompted with a synthetic test string ("What is 2+2?") and the output
was inspected for correctness. The configurations are listed in
section 3 in chronological order of execution. Successful runs report
weight loading time, engine initialisation time, KV cache allocation,
and tokens-per-second on cold first inference. Failed runs report the
specific error mode.

The following environment was set for all runs:

```bash
unset PYTORCH_ALLOC_CONF                 # expandable_segments unsupported on ROCm
export VLLM_ROCM_USE_AITER=0             # AITER kernels unstable on gfx1201
export AMD_SERIALIZE_KERNEL=1            # NOT 3 — newer PyTorch rejects it silently
export HIP_LAUNCH_BLOCKING=1
```

Measurements were taken from a single Python process per configuration,
with the vLLM virtual environment (`~/venvs/vllm/`) freshly activated.
No GPU warmup was performed prior to the timed inference.

## 3. Results

### 3.1 Configuration 1 — FP8 TP=1 default: out-of-memory

```python
LLM(model='qwen36-27b-fp8', tensor_parallel_size=1,
    max_model_len=4096, gpu_memory_utilization=0.85)
```

Result: `torch.OutOfMemoryError: HIP out of memory. Tried to allocate
180.00 MiB. GPU 0 has a total capacity of 31.86 GiB of which 54.00 MiB
is free.` The model weights occupy 29.05 GiB after `_maybe_pad_fp8_weight()`
rounds tensors for FP8 alignment. With `gpu_memory_utilization=0.85`,
the budget is 27.2 GiB — already exceeded by weights alone before any
KV cache or activations can be allocated.

### 3.2 Configuration 2 — FP8 TP=2 default CUDA graphs: HSA crash

Adding `tensor_parallel_size=2` to share weights across both GPUs.
Initialisation succeeded: 15.35 GiB per GPU for weights, 8.94 GiB for
KV cache, 5.55 GiB for CUDA graphs, total init time 252 s. At first
inference:

```
:0:rocdevice.cpp :3586: Callback: Queue aborting with error :
HSA_STATUS_ERROR_INVALID_PACKET_FORMAT: The AQL packet is malformed.
torch.AcceleratorError: HIP error: unspecified launch failure
```

CUDA graph capture itself completed without diagnostic; graphs crashed
on replay. The hybrid Mamba + Transformer attention used by Qwen
3.5/3.6 family appears to trigger an AQL packet formatting issue
specific to gfx1201 + ROCm 7.2.1.

A side-finding emerged in the same log:

```
UserWarning: Ignoring invalid value for boolean flag
AMD_SERIALIZE_KERNEL: 3 valid values are 0 or 1.
```

Newer PyTorch only accepts `AMD_SERIALIZE_KERNEL` as `0` or `1`. The
previously circulated workaround `=3` is silently ignored, leaving the
kernel-serialisation behaviour at its default.

### 3.3 Configuration 3 — FP8 TP=2 with `enforce_eager=True`: working

```python
LLM(model='qwen36-27b-fp8', tensor_parallel_size=2,
    max_model_len=2048, gpu_memory_utilization=0.85,
    enforce_eager=True)
```

Result: PASSED.

| Metric | Value |
|---|---|
| Weight loading | 3.69 s |
| Total model load | 4.06 s, 15.35 GiB / GPU |
| Engine init | 24.33 s |
| KV cache available | 7.53 GiB / GPU |
| KV cache size | 61 152 tokens |
| Max concurrency at 2 K context | 52.33× |
| First inference (cold) | 4.82 s for 20 tokens (4.15 tok/s) |

Output: `'\n\n2 + 2 = 4.\n\nWhat is 3 times 4?\n\n3'` — correct
decoding, base-model continuation behaviour.

### 3.4 Configuration 4 — BF16 TP=2 with `util=0.85`: KV cache underflow

```python
LLM(model='qwen36-27b', tensor_parallel_size=2,
    max_model_len=2048, gpu_memory_utilization=0.85,
    enforce_eager=True)
```

Result: model loaded (25.76 GiB / GPU, 17.18 s) but engine init failed:

```
Available KV cache memory: -0.5 GiB
ValueError: No available memory for the cache blocks.
```

With `util=0.85` and 32 GB cards, the budget is 27.2 GiB. BF16 weights
took 25.76 GiB, framework overhead consumed the remainder, leaving a
0.5 GiB deficit for KV cache.

### 3.5 Configuration 5 — BF16 TP=2 with `util=0.95`, `max_len=1024`: working

Tightening memory budget by raising utilisation and halving context
length.

Result: PASSED.

| Metric | Value |
|---|---|
| Weight loading | 5.91 s |
| Total model load | 6.13 s, 25.76 GiB / GPU |
| Engine init | 19.53 s |
| KV cache available | 2.68 GiB / GPU |
| KV cache size | 7 056 tokens |
| Max concurrency at 1 K context | 7.20× |
| First inference (cold) | 2.77 s for 20 tokens (7.23 tok/s) |

Output: identical to Configuration 3 (`'\n\n2 + 2 = 4.\n\nWhat is 3
times 4?\n\n3'`), confirming numerical agreement between the two
working configurations on this prompt.

### 3.6 Working configurations summary

| Variant | TP | enforce_eager | max_len | util | Weights/GPU | KV cache | Cold tok/s |
|---|---|---|---|---|---|---|---|
| FP8 | 2 | true | 2048 | 0.85 | 15.35 GiB | 7.53 GiB | 4.15 |
| BF16 | 2 | true | 1024 | 0.95 | 25.76 GiB | 2.68 GiB | 7.23 |

The two working configurations represent different operating points
on a context-length vs throughput trade-off: FP8 buys roughly 8.7× the
KV cache (61 K vs 7 K tokens) at the cost of approximately 43% lower
cold throughput.

## 4. Discussion

### 4.1 The BF16 > FP8 inversion

The community expectation, formed largely on H100 and MI300X
benchmarks, is that FP8 inference outperforms BF16 by roughly 1.5–2×
on equivalent prompts, courtesy of doubled effective memory bandwidth
and dedicated tensor-core paths. On the present hardware and software
stack, the inverse holds — BF16 is approximately 75% faster on cold
first inference.

The vLLM logs reveal the mechanism:

```
WARNING fp8_utils.py:1185 Using default W8A8 Block FP8 kernel config.
Performance might be sub-optimal! Config file not found at
.../configs/N=8192,K=5120,device_name=AMD_Radeon_AI_PRO_R9700,
dtype=fp8_w8a8,block_shape=[128,128].json
```

vLLM ships pre-tuned FP8 kernel configurations for established AMD
silicon (the MI300X family in particular) but not yet for R9700 /
gfx1201. The fallback is a generic block-FP8 path, which on this
hardware is substantially slower than the native ROCm BF16 path.

This finding is **stack-specific, not architecture-specific**. The
inversion is a property of vLLM 0.19.0's kernel coverage in April
2026, not of the R9700 hardware itself. A future vLLM release with
R9700-tuned FP8 kernels, or a different runtime altogether
(e.g. llama.cpp with Q4_K_M), may exhibit the conventional ordering.
The implication for community guidance, however, is concrete: until
those kernel configurations exist, BF16 is the recommended starting
point on this hardware for users prioritising tokens-per-second over
context window.

### 4.2 Cross-reference to CUDA llama.cpp

For order-of-magnitude calibration, an independently published
benchmark of the same model class on NVIDIA hardware was reviewed. A
public benchmark (Digital Spaceport, YouTube, 2026-04-23) using
llama.cpp with Q4_K_M quantisation reports the following steady-state
text-generation throughput:

| Hardware | Backend | Quantisation | TG @ 512 |
|---|---|---|---|
| 1× RTX 3090 24 GB | llama.cpp | Q4_K_M | 39 tok/s |
| 1× RTX 4090 24 GB | llama.cpp | Q4_K_M | 45.81 tok/s |
| 2× RTX 3090 | llama.cpp | Q4_K_M | ~46 tok/s |
| 2× RTX 4090 | llama.cpp | Q4_K_M | 45.89 tok/s |

These numbers are not directly comparable to the cold first-inference
measurements in section 3 — they reflect (a) steady-state rather than
cold runs, (b) CUDA graph paths versus eager mode, (c) Q4_K_M
quantisation with a smaller working set than FP8 or BF16. They are
included only to establish that the present cold-inference
measurements (4.15–7.23 tok/s) are within an order of magnitude of
what is achievable for this model class on broadly comparable hardware.
A direct apples-to-apples comparison requires the steady-state sweep
identified in section 5.

### 4.3 Practical guidance for RDNA 4 users

Three concrete recommendations emerge from the present study, with the
caveat that they apply specifically to vLLM 0.19.0 + ROCm 7.2.1 +
gfx1201 in April 2026:

1. **Use TP=2 for any 27B-class model**, regardless of quantisation.
   Single-GPU configurations leave no KV cache headroom on 32 GB cards.

2. **Set `enforce_eager=True` for any model in the Qwen 3.5 / 3.6
   family.** CUDA-graph capture succeeds during initialisation but
   crashes on first replay.

3. **Set `AMD_SERIALIZE_KERNEL=1`, not `3`.** The previously
   circulated value `3` is silently ignored under newer PyTorch.

These recommendations are likely to evolve. ROCm point-release
updates may resolve the CUDA-graph issue. vLLM nightly builds may
ship R9700-tuned FP8 kernels that close the FP8 → BF16 inversion gap.

## 5. Limitations

This study is deliberately scoped to first-inference characterisation
on a single workstation. Several limitations apply.

**Cold first-inference only.** All measurements are cold first
inference of a single 20-token continuation. A proper steady-state
throughput sweep with N ∈ {10, 25, 50, 100, 200, 500} concurrent
requests, varied input/output lengths, and warm-state plateaus is the
direct successor to this work and was not in scope for the v0.1.0
release.

**Single workstation, single sample.** All measurements come from one
physical workstation. Hardware-level variance (binning, thermal
behaviour, board partner), motherboard chipset effects, and PCIe
topology have not been characterised. Replication on independent
R9700 systems is welcomed.

**Single model family.** The findings on Qwen 3.6 27B may or may not
generalise to other 27B-class models. The hybrid Mamba + Transformer
attention pattern of the Qwen 3.5 / 3.6 family is the suspected cause
of the CUDA-graph crash; pure-Transformer models in the same parameter
range may not require `enforce_eager=True`.

**Synthetic prompt only.** The inference correctness check used a
single synthetic arithmetic prompt. This validates that decoding
proceeds and that output text is syntactically reasonable; it does not
characterise generation quality, perplexity, or downstream task
performance. Quality-aware evaluation is out of scope for a
hardware-envelope study.

**AI-assisted documentation.** The experimental subject of this
study is local LLM deployment via vLLM and ROCm; the documentation of
that subject was prepared with assistance from cloud-based AI tools
(Claude, GPT). The technical findings are hardware/software-empirical
and do not depend on the documentation tooling used to write them up.
For the present benchmark, no sensitive data was involved — only
synthetic test prompts and openly licensed model weights downloaded
from Hugging Face — so cloud-LLM-assisted documentation introduces no
confidentiality concern. Future work involving sensitive data would
require local-only tooling end-to-end. Full disclosure of AI tool
usage is in `AI_USAGE_DISCLOSURE.md` in the project repository.

## 6. Conclusion

We have documented the practical envelope of running Qwen 3.6 27B on
a dual-R9700 workstation under vLLM 0.19.0 + ROCm 7.2.1. Two
configurations reach inference, both requiring `tensor_parallel_size=2`
and `enforce_eager=True`. Under the current software stack, BF16
outpaces FP8 by approximately 75% on cold first inference, an
inversion of the typical datacenter-benchmark expectation, traceable
to missing R9700-specific FP8 kernel configurations in vLLM. The
mandatory environment variables and configuration choices have been
stated explicitly to enable reproduction. Steady-state characterisation
on a concurrency sweep is the natural successor study.
